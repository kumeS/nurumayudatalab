2.2 CSVファイル処理フロー
------------------------------------------------------------------------

[ファイルドロップ or ファイル選択]
  │
  └─ handleFiles(files)
       │
       ├─ DB初期化完了を待機（最大10秒、100msポーリング）
       │
       ├─ CSVファイルのみフィルタリング
       │
       ├─ ファイル名の日付でソート（inferWeekFromDate使用）
       │
       └─ 各ファイルについて:
            │
            ├─ generateFileHash(file) [SHA-256]
            │    Web Crypto APIでファイル内容のハッシュ生成
            │
            ├─ checkDuplicateFile(hash)
            │    IndexedDB fileDataストアのhashインデックスで検索
            │    + uploadedFiles Mapでファイル名チェック
            │
            ├─ inferWeekFromDate(fileName) [utils.js]
            │    正規表現でファイル名からYYMMDD-YYMMDD形式を解析
            │    → { date: 終了日, startDate: 開始日, duration: 期間日数 }
            │
            ├─ parseCSVFile(file, weekInfo) [data.js]
            │    FileReader.readAsText(file, 'UTF-8')
            │    Papa.parse(content, { header: true, ... })
            │    ヘッダー改行除去/トリム
            │    空ASIN/タイトルの行を除外
            │    数値正規化: ￥/,/% を除去してパース
            │    → { asin, title, sessions, pageViews, orders, conversionRate,
            │        sales, orderItems, weekDate, duration, fileName }
            │
            ├─ saveFileToIndexedDB(file, weekDate, hash, data)
            │    3つのストアに同時書き込み:
            │    fileData: ファイル情報+生CSV内容
            │    amazonData: パース済みレコード（fileHash付き）
            │    uploadHistory: アップロード履歴
            │
            └─ amazonData配列に追加 + uploadedFiles Mapに追加

       [全ファイル処理後]
       └─ updateFileList() + updateDashboard()


2.3 データ分析・分類パイプライン
------------------------------------------------------------------------

[updateDashboard()] [ui.js]
  │
  ├─ updateAsinTracking() [data.js] ← 核心のデータ処理
  │    │
  │    ├─ _.groupBy(amazonData, 'asin') でASIN別グルーピング
  │    │
  │    └─ 各ASINについて:
  │         │
  │         ├─ Step 1: 日割り計算 + 日次マップ構築
  │         │    各レコードのsessions/sales/orders/pageViewsを
  │         │    duration（期間日数）で割り、開始日〜終了日の各日に割当
  │         │    dailyMap: Map<YYYY-MM-DD, {sessions, sales, ...}>
  │         │
  │         ├─ Step 2: 重複解決 (Overlap Resolution)
  │         │    同一日に複数ファイルからデータがある場合:
  │         │    - 短い期間のファイルを優先（粒度優先）
  │         │    - 同期間の場合は累積平均
  │         │
  │         ├─ Step 3: 線形補間 (Linear Interpolation)
  │         │    連続する2点間の欠損が7日以内の場合、線形補間で埋める
  │         │    y = y1 + (y2 - y1) * (x - x1) / (x2 - x1)
  │         │    8日以上の欠損はデータなしとして扱う
  │         │    補間データにはisInterpolated=true, duration=0をマーク
  │         │
  │         ├─ Step 4: 季節性判定
  │         │    determineSeasonality(title): タイトルのキーワードで判定
  │         │    SS: 春/夏/SS/半袖/サマー → 春夏（3-8月がオンシーズン）
  │         │    AW: 秋/冬/AW/長袖/ウインター → 秋冬（9-2月がオンシーズン）
  │         │    ALL: 通年/オールシーズン or デフォルト
  │         │    補助マスタによるオーバーライド対応
  │         │
  │         ├─ Step 5: 統計値算出
  │         │    totalSessions, totalSales, maxSessions
  │         │    日次CVR: (orders / sessions) * 100
  │         │    AOV: sales / orders
  │         │    RPS: sales / sessions
  │         │
  │         ├─ Step 6: 変化率計算（7日チャンク比較）
  │         │    末尾から7日ずつ区切り、前週比セッション変化率を算出
  │         │    avgChangeRate（平均変化率）、volatility（標準偏差）
  │         │
  │         ├─ Step 7: ステータス判定 [determineAsinStatus()]
  │         │    new: 初出から4週以内
  │         │    surging: 直近変化率 > 50%
  │         │    declining: 直近変化率 < -30%
  │         │    growing: 平均変化率 > 20%
  │         │    shrinking: 平均変化率 < -15%
  │         │    volatile: ボラティリティ > 30
  │         │    stable: その他
  │         │
  │         └─ Step 8: 商品分類 [classifyProduct()]
  │              直近30日データを集約し、以下の優先順で排他分類:
  │
  │              1. カテゴリI: 急落警戒
  │                 条件: 売上 or セッションが前期間比30%以上減少
  │                        かつ季節オフでない
  │
  │              2. カテゴリD: 改善困難
  │                 条件: セッション >= 300 かつ CVR <= 1%
  │
  │              3. カテゴリG: 高単価・機会損失
  │                 条件: AOV > 平均の1.5倍 かつ CVR < 1% かつ セッション > 200
  │
  │              4. カテゴリC: 広告強化推奨
  │                 条件: セッション > 200 かつ CVR >= 5%
  │
  │              5. カテゴリA: 新規商品
  │                 条件: 初出から4週以内
  │
  │              6. カテゴリH: 合わせ買い推奨
  │                 条件: AOV < 平均の0.5倍 かつ CVR > 5%
  │
  │              7. カテゴリB: 自力成長期待
  │                 条件: セッション > 200 かつ 3% <= CVR < 5%
  │
  │              8. カテゴリK: 販売継続
  │                 条件: セッション > 100 かつ CVR >= 3%
  │
  │              9. カテゴリE: セッション不足
  │                 条件: セッション < 100
  │
  │              10. カテゴリF: 注文不足
  │                  条件: 注文 < 10
  │
  │              11. カテゴリJ: 標準（いずれにも該当しない場合）
  │
  ├─ updateStats() [ui.js]
  │    総売上、アクティブASIN数、新規ASIN数、平均成長率を算出・表示
  │    前週比トレンド（↑↓）の表示
  │
  ├─ generateAlerts() → displayAlerts() [ui.js]
  │    新商品好調/セッション急上昇/急落のアラートカード生成
  │    最大5件表示
  │
  └─ updateView(activeView) [ui.js]
       現在アクティブなタブに応じたビュー描画


2.4 ビュー切替・表示ロジック
------------------------------------------------------------------------

updateView(viewName) がハブとなり、各ビューを切替:

  'monitoring'    → displayAsinMonitoring()
                     ASINカードをステータス順にソート、上位12件表示
                     各カードにスパークライン描画
                     クリックでモーダル詳細チャート表示

  'timeline'      → createTimelineChart()
                     売上上位20%のASINを抽出
                     全日次データからユニーク日付をX軸に
                     ASINごとのセッション数を折れ線グラフで描画

  'anomaly'       → displayAnomalyDetection()
                     変化率30%超のASINを抽出
                     createAnomalyChart()で棒グラフ描画
                     上位6件をASINカードとして表示

  'newproducts'   → displayNewProducts()
                     status='new'のASINを抽出
                     初週/翌週セッション比較、成長率算出
                     createNewProductsChart()で複合チャート描画

  'classification'→ displayClassification()
                     カテゴリ別件数サマリー表示（カラーバッジ）
                     フィルタチップでカテゴリ絞り込み
                     分類カード表示（カテゴリ色、判定理由、推奨アクション）

  'effect'        → displayEffectAnalysis()
                     ★ 空関数（未実装）
                     UIにはdateInputとanalysisボタンが存在

  'advanced'      → displayAdvancedAnalysis()
                     CVR分析（棒グラフ）、効率分析（棒グラフ）、
                     パフォーマンスマトリックス（散布図）の3チャート

4. 未実装だが有用な新機能の提案
================================================================================

以下はすべて短期的に実装可能（数時間〜数日）で、大規模な本番環境を
必要としない提案です。

■ 4.1 施策効果測定（Before/After比較）の実装 [優先度: 高]
------------------------------------------------------------------------
  概要: 既にUIが存在する effectView の中身を実装する。
  実装内容:
    - 基準日を入力すると、全ASINについて基準日前後の平均指標を比較
    - 各ASINのBefore/After変化率を棒グラフ + テーブルで表示
    - 「改善」「悪化」「変化なし」のサマリーカウント
  工数目安: 3〜5時間
  理由: UIの骨格はある。classifyProduct()と同様のデータ集約ロジックを
        応用すれば短期間で実装できる。実務的な価値が高い。

■ 4.2 パレート分析（ABC分析）[優先度: 高]
------------------------------------------------------------------------
  概要: 売上貢献度に基づくABC分類を追加する。
  実装内容:
    - 全ASINを売上降順にソートし、累積売上比率を計算
    - A（上位80%）、B（次の15%）、C（残り5%）に分類
    - 既存の商品分類ビューにABCランクを追加表示
    - パレート曲線（折れ線 + 棒の複合チャート）を描画
  工数目安: 2〜4時間
  理由: 仕様書の「異常検知・ランキング」セクションで言及済み。
        既存のCharts.jsインフラを活用して短期実装可能。

■ 4.3 ASINカードの30日集約値表示 [優先度: 高]
------------------------------------------------------------------------
  概要: モニタリングカードの指標を「最新1日分」から「直近30日集約」に変更。
  実装内容:
    - asinTracking に last30DaysStats を追加計算
    - createAsinCard() で集約値を表示
    - 月間セッション数/注文数/売上/CVRを明確に表示
  工数目安: 1〜2時間
  理由: 現在の日割り表示は直感的でなく、仕様との乖離がある。
        既存データ構造を活用した小さな変更で大きくUXが改善する。

■ 4.4 返金率の取り込みと全体警告表示 [優先度: 中]
------------------------------------------------------------------------
  概要: CSVから返金率を取り込み、全体返金率の加重平均を算出・警告表示。
  実装内容:
    - parseCSVFile()に返金率フィールドの取得を追加
    - calculateGlobalStats()で加重平均返金率を算出
    - ダッシュボードの統計カードに「全体返金率」を追加
    - 5%超の場合に警告アラート表示
  工数目安: 2〜3時間
  理由: 仕様書の明確な要件であり、既存のパーサーとUI枠組みに
        追加するだけで実装可能。

■ 4.5 推定値・データ品質インジケーター [優先度: 中]

    - ダッシュボード全体にデータカバレッジ率の表示

■ 4.6 閾値カスタマイズUI [優先度: 中]

  概要: 分類に使う閾値（セッション数、CVR%等）をUI上で変更可能にする。

    - 変更後に再分類を実行

  理由: 仕様書Phase 2で「UI化」が予定されているが、現状の定数管理では

================================================================================
5. 技術的注意事項・既知の課題
================================================================================

■ グローバル変数への依存
  バンドル版ではIIFE内にすべてのコードが閉じ込められるが、
  window.filterClassification / window.runEffectAnalysis が
  HTMLのonclick属性から呼び出されるため、グローバルスコープへの
  明示的な公開が必要。現在は main.js で設定済み。

■ dynamic import の残存 (main.js)
  clearDataBtn のハンドラ内に `await import('./db.js')` が残っている。
  バンドル版では動作しない可能性がある（clearAllDataは同一スコープ内に
  存在するため直接呼べるが、importが失敗するリスクあり）。

■ デバッグログの過多
  main.js に大量の console.log が残存。本番環境では不要。
  開発完了後に削除またはログレベル制御の導入が望ましい。

■ テストの完全欠如
  ユニットテスト/統合テストが一切存在しない。
  特に classifyProduct()、updateAsinTracking()、inferWeekFromDate()
  などのコアロジックはテストがあるべき。

■ エラーハンドリングの一部不足
  IndexedDBのトランザクションエラー時のリカバリーが不十分。
  CSVパース失敗時のユーザーフィードバックが汎用的。
