================================================================================
【1】OB-1 の概要
================================================================================

■ 何者か
  - ソフトウェアエンジニアリングに特化した対話型 CLI エージェント
  - テキスト指示だけでコード読み書き・実行・検索・ブラウザ操作を自律実行
  - 単一モデルではなく、複数の専門サブエージェントを組み合わせて動作
  - ローカル環境で完結（外部サーバー不要のタスクも多い）

■ 基本的な動き
  - ユーザーが自然言語で指示 → OB-1 がツールを選択・実行 → 結果を返す
  - 必要に応じてサブエージェントに委譲し、並列・直列に処理
  - 確認が必要な操作（ファイル変更・コマンド実行）はユーザーに承認を求める
  - セッションをまたいで記憶を保存・復元できる

■ 対応できる主なタスク
  - コードの読解・バグ修正・新機能実装・リファクタリング
  - ファイル作成・編集・削除・検索
  - シェルコマンドの実行・自動化スクリプトの作成
  - Web 検索・URL コンテンツの取得・要約
  - ブラウザ操作・スクリーンショット撮影・UI テスト
  - 画像・動画の内容解析・OCR
  - コードベース全体の構造把握・アーキテクチャ分析
  - ドキュメント生成・レポート作成

■ 技術スタック
  - モデル: Anthropic Claude / OpenAI GPT / Google Gemini 等、複数切り替え可能
  - ツール: ファイルI/O・シェル・grep・glob・Web検索・ブラウザ（Playwright）
  - 記憶: セッション内メモリ + ディスク永続化（~/.ob1/memory/）
  - 外部連携: Braintrust / PostHog / Linear / Sentry


================================================================================
【2】重要なポイント
================================================================================

■ コンテキストの消費を意識する
  - 会話が長くなるほどコンテキストが圧迫され、後半の指示精度が落ちる
  - 大きなタスクはサブエージェント（worker など）に委譲してメインを軽量に保つ
  - Web 検索・画像読込みはそれぞれ専用エージェント経由で隔離する
  - 長期プロジェクトは定期的に /memory save でセッションを保存する

■ 承認フローを理解する
  - ファイル変更・シェルコマンドなど副作用のある操作はユーザー承認が必要
  - キャンセルしたツール呼び出しは OB-1 が再度試みることはない
  - 「やること / やらないこと」を明示するとスコープ外の変更を防げる

■ エージェントの使い分けが品質を左右する
  - 間違ったエージェントを使うと遅い・精度が低い・副作用が出る
  - explore は読み取り専用（変更不可）・worker は変更可能、という差が大きい
  - 画像は必ず vision-analyzer 経由・Web は必ず web 経由（原則）

■ モデルは目的に応じて切り替える
  - デフォルト（Claude Sonnet）で解決できないときは opus / gemini-2.5-pro へ
  - 軽量タスクは haiku / gemini-2.5-flash でコストを下げられる
  - サブエージェント呼び出し時にモデルを個別指定できる

■ コミット・破壊的操作は明示的指示のみ
  - 「コミットして」と言わない限り git commit は実行しない
  - git reset --hard / force push などは明示的に指示した場合のみ
  - 本番コードの大きな変更前は /thread fork でバックアップを作ると安全

■ プロジェクト固有の情報はグローバル記憶に保存しない
  - save_memory（グローバル）はユーザー個人の好み・設定向け
  - ファイルパス・テストコマンドなどプロジェクト固有情報はプロジェクト内文書に


================================================================================
【3】活用の利点
================================================================================

■ 開発速度の向上
  - 調査・実装・確認を一連で自律実行するため往復コストが大幅に減る
  - ファイルを手動で開いて読む作業を OB-1 に任せられる
  - バンドル生成・grep 確認・ログ確認などの定型作業を自動化できる

■ 品質の向上
  - 既存コードの規約・スタイルをコード読解してから変更するため一貫性が保たれる
  - テスト作成・バグ確認・動作確認まで一連でこなせる
  - エラーメッセージ・スタックトレースを直接渡せるため診断精度が高い

■ 知識の補完
  - 未知のライブラリ・API は web エージェントで即時調査してから実装に使える
  - コードベースに不慣れでも codebase_investigator で全体像を把握できる
  - ドキュメント・引き継ぎ資料を自動生成できるため属人化を防げる

■ コンテキストの保存と再利用
  - セッションを保存・再開することで長期プロジェクトを継続できる
  - スレッドのフォークで「仮説検証」を安全に行える
  - 過去の会話を thread ツールで参照・分岐できる

■ コスト効率
  - 軽量タスクにはコストの低いモデルを割り当てられる
  - サブエージェントへの委譲でメインモデルの呼び出し回数を抑えられる
  - 一度の指示で複数ファイルの変更・確認まで完了するため人間の作業時間を削減

■ 安全性
  - 副作用のある操作はすべてユーザー承認が必要（誤操作を防ぐ）
  - セキュリティ上の問題（APIキーの露出など）があれば警告する
  - /thread fork でリスクゼロの実験環境を作れる


================================================================================
【4】使い方
================================================================================

■ 基本的な指示の出し方
  - 自然言語で指示するだけでよい（コマンド構文は不要）
  - 「ゴール → 制約 → 背景」の順で書くと精度が上がる
  - ファイル名・関数名など具体的なシンボルを含めると精度が上がる
  - 「やること」と「やらないこと」を分けて書くとスコープが明確になる
  - エラーは省略せずにメッセージ全文を貼る

■ スラッシュコマンドの使い方
  - /help               → コマンド一覧を確認する
  - /bug                → バグ・フィードバックを報告する
  - /plan start <goal>  → 多段階タスクを自動実行する
  - /plan status        → 実行中プランの進捗を確認する
  - /plan abort         → 実行中プランを中断する
  - /thread new         → プロジェクト別に新しいスレッドを作る
  - /thread list        → 過去のスレッド一覧を見る
  - /thread switch <id> → 別のスレッドに切り替える
  - /memory save <id>   → セッションをディスクに保存する
  - /memory load <id>   → 保存済みセッションを復元する
  - /model set <key>    → 使用モデルを切り替える
  - /agents list        → 利用可能なエージェント一覧を確認する
  - /agents refresh     → エージェント定義を再読み込みする
  - /config show        → 現在の設定を確認する

■ サブエージェントの使い分け
  - worker              → コード実装・ファイル変更・コマンド実行（最頻用）
  - explore             → コードの調査・検索（読み取り専用）
  - codebase_investigator → プロジェクト全体の構造把握・バグ根本原因分析
  - plan                → 実装計画の立案（実行はしない）
  - general             → 調査しながら変更も必要な複合タスク
  - browser             → Webページの操作・スクリーンショット撮影
  - vision-analyzer     → 画像・動画・スクリーンショットの内容解析（必須経由）
  - web                 → Web 検索・URL コンテンツ取得（必須経由）
  - cli_help            → OB-1 自身の機能・使い方の確認

■ モデルの選び方
  - claude-sonnet-4.6   → 通常のコーディング・実装（デフォルト、これで十分）
  - opus / opus-4.6     → 複雑な設計・難しいバグ・高精度な推論が必要な場合
  - gemini-2.5-pro      → 深い推論・大量コンテキスト処理が必要な場合
  - haiku               → 単純な変換・フォーマット・軽量タスク（高速・安価）
  - gemini-2.5-flash    → 高速処理が必要な軽量タスク
  - deepseek-r1-free    → コスト最優先のとき（無料枠）

■ スレッド・記憶の使い方
  - プロジェクトごとに /thread new でスレッドを分ける
  - 長期作業は /memory save で都度保存して再開時に /memory load
  - 別の方向性を試したいときは /thread fork で分岐してから実験する
  - 「〇〇を覚えておいて」でユーザー個人の好み・設定をグローバル保存できる

■ 実装タスクの進め方（推奨フロー）
  Step 1: codebase_investigator または explore で現状把握
  Step 2: plan で実装計画を確認（大きな変更の場合）
  Step 3: worker に実装を委譲（ファイル変更・コマンド実行を含む）
  Step 4: browser でブラウザ確認 / vision-analyzer でスクリーンショット確認
  Step 5: 問題があれば worker に修正を委譲

■ バグ修正の進め方
  Step 1: 症状・再現手順・エラーメッセージを正確に伝える
  Step 2: codebase_investigator または general で根本原因を特定
  Step 3: worker に修正を委譲
  Step 4: 同じ手順で再現しないことを確認

■ データ分析タスクの進め方
  Step 1: データの形式・分析目的・出力形式を明示して指示
  Step 2: worker が分析スクリプトを作成・実行
  Step 3: vision-analyzer で出力グラフを確認
  Step 4: 追加の深掘り分析を指示

■ よくある失敗と対策
  - 指示が曖昧でスコープ外を変更された  → やること/やらないことを明示する
  - コンテキストが長くなって精度が落ちた → worker に委譲してメインを軽量化
  - 画像の内容を正しく読めなかった       → 必ず vision-analyzer 経由にする
  - Web 検索の結果がコンテキストを汚染   → 必ず web エージェント経由にする
  - バンドル生成を忘れて変更が反映されない → 実装指示に「バンドル再生成も」と添える
  - 前のセッションの文脈が引き継がれない  → 毎セッション冒頭で背景・進捗を簡潔に伝える
  - 複数の無関係タスクを一度に渡した      → 1 指示 1 タスクを基本にする
