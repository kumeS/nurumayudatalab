================================================================================
PART 1: コンテキストエンジニアリングの基礎
================================================================================

コンテキストエンジニアリングとは、OB-1 に渡す情報の「何を・どの順で・
どの粒度で」伝えるかを設計する技術です。指示の書き方ひとつで、
アウトプットの質・速度・コストが大きく変わります。

────────────────────────────────────────────────────────────────────────────────
原則1: ゴール → 制約 → コンテキスト の順で伝える
────────────────────────────────────────────────────────────────────────────────

  ❌ 悪い例（曖昧・順序が逆）:
  「data.js を見て、なんか問題があれば直して。あと仕様書も読んで。」

  ✅ 良い例（ゴール先行・制約明示・コンテキスト補足）:
  「data.js の parseCSVFile() に返金率（refundRate）フィールドの取り込みを
   追加してください。
   【制約】既存のフィールド定義（sessions, orders, sales 等）の形式に揃えること。
   【背景】現在 CSV の "返金率" カラムを読み飛ばしており、README.md の
   仕様書 2.2 節に定義されたカラムが未取得の状態です。」

────────────────────────────────────────────────────────────────────────────────
原則2: 調査と実装を分離する
────────────────────────────────────────────────────────────────────────────────

  調査フェーズと実装フェーズを一度の指示に混ぜると、
  OB-1 が探索と変更を同時に行い、意図しない変更が発生しやすくなります。

  推奨フロー:
    Step 1: explore / codebase_investigator で現状把握（読み取り専用）
    Step 2: 把握した結果をもとに実装指示を出す
    Step 3: worker に実装を委譲

────────────────────────────────────────────────────────────────────────────────
原則3: サブエージェントに委譲してコンテキストを温存する
────────────────────────────────────────────────────────────────────────────────

  大きなタスクをメインセッションで直接こなすと、会話コンテキストが
  長大になり、後半の指示精度が落ちます。

  目安:
    - ファイル数 3 個以上の変更 → worker に委譲
    - Web 調査が必要 → web エージェント経由
    - 画像・スクリーンショットの確認 → vision-analyzer 経由
    - コードベース全体の理解が必要 → codebase_investigator を先に走らせる

────────────────────────────────────────────────────────────────────────────────
原則4: 「なぜ」を添える
────────────────────────────────────────────────────────────────────────────────

  OB-1 は「なぜ変更するか」を知ることで、より文脈に合った実装を選択します。

  ❌ 「classifyProduct() の条件を変えて」
  ✅ 「classifyProduct() のカテゴリ E の閾値を 100 → 150 に変更してください。
      理由: 実運用データで 100〜150 セッションの商品が誤って露出不足と
      判定される事例が多発しており、閾値を引き上げて精度を改善するためです。」

タスク例 B: バグ調査と修正（ASINカードの数値がおかしい）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【背景】
モニタリングビューの ASIN カードに「セッション: 3.23」のような
小数値が表示されている。月間合計のはずが日割り値が出ている模様。

─── プロンプト（一発で完結する場合）────────────────────────────────────────

  「ASINモニタリングカードのセッション数が小数で表示されるバグを調査・修正してください。
   【症状】
   モニタリングビューのカードに「セッション: 3.23」のような小数が表示される。
   本来は直近30日の合計セッション数（整数）を表示すべき。
   【調査ポイント】
   ui.js の createAsinCard() が weeklyData からどの値を取得しているか確認。
   weeklyData の実体が日次データ（sessions = 期間合計 ÷ 日数）であることが原因の可能性。
   【修正方針候補】
   1. asinTracking に last30Days 集計値を追加して、それをカードに表示
   2. weeklyData の最新値を表示している箇所で ×30 するだけ（精度は低い）
   方針1で修正し、バンドル再生成まで行ってください。」

  ← 調査と修正を一発で任せてよいケース。
    症状・調査ポイント・修正方針まで書いているため worker が迷わない。

   1. ASIN別の月間売上（期間重複は期間が短いファイルを優先して按分）

PART 3: コンテキストエンジニアリング テクニック集
================================================================================

────────────────────────────────────────────────────────────────────────────────
テクニック 1: ファイル・関数名を「錨（アンカー）」として使う
────────────────────────────────────────────────────────────────────────────────

  OB-1 はファイル名・関数名を手がかりにコードを読みます。
  あいまいな表現より具体的なシンボル名を使うと精度が上がります。

  ❌ 「分類のロジックを直して」
  ✅ 「js/data.js の classifyProduct() 内の、カテゴリ E の判定条件を修正して」

────────────────────────────────────────────────────────────────────────────────
テクニック 2: 期待する出力の形式を先に示す
────────────────────────────────────────────────────────────────────────────────

  何を返してほしいかを先に書くと、OB-1 がゴールから逆算して作業します。

  例:
  「以下の形式で分析結果を返してください:
   【完成度】XX%
   【変更ファイル】...
   【動作確認方法】...
   の形式で最後にまとめを出力してください。」

────────────────────────────────────────────────────────────────────────────────
テクニック 3: 「やること」と「やらないこと」を明示する
────────────────────────────────────────────────────────────────────────────────

  スコープを絞ることで不要な変更を防ぎます。

  例:
  「【やること】
   ui.js の displayEffectAnalysis() にロジックを追加する
   【やらないこと】
   - index.html の HTML 構造を変更しない（既に effectView セクションがある）
   - 新しい依存ライブラリを追加しない
   - main.js・db.js は触らない」

────────────────────────────────────────────────────────────────────────────────
テクニック 4: 段階的に委譲する（分割統治）
────────────────────────────────────────────────────────────────────────────────

  大きなタスクを一発で渡すより、段階的に渡す方が品質が上がります。

  大きなタスク（悪い例）:
  「施策効果測定機能を設計して実装してテストして動作確認まで全部やって」

  分割した例（良い例）:
  Turn 1: 「現状を調査して設計方針を提案して」（← explore/plan）
  Turn 2: 「提案した方針で実装して」（← worker）
  Turn 3: 「実装後にブラウザで動作確認のスクリーンショットを撮って」（← browser）
  Turn 4: 「スクリーンショットで問題があれば修正して」（← worker）

────────────────────────────────────────────────────────────────────────────────
テクニック 5: エラーメッセージをそのまま貼る
────────────────────────────────────────────────────────────────────────────────

  バグ修正の際はエラーメッセージ・スタックトレースを省略せずに貼ります。

  例:
  「以下のエラーが発生しています。修正してください。
   ---
   Uncaught TypeError: Cannot read properties of undefined (reading 'sessions')
       at createAsinCard (bundle-static.js:1823:42)
       at displayAsinMonitoring (bundle-static.js:1798:26)
   ---
   再現手順: CSVをアップロード後、ASINモニタリングタブをクリック」

────────────────────────────────────────────────────────────────────────────────
テクニック 6: 「確認してから進めて」パターン
────────────────────────────────────────────────────────────────────────────────

  大きな変更の前に、OB-1 の理解が正しいかを確認させます。

  例:
  「実装を始める前に、以下を確認してください:
   1. asinTracking の dailyData の構造を read_file で確認して
   2. 変更する関数の現在のコードを表示して
   3. 実装方針を一行で説明してから、進めてください。」

────────────────────────────────────────────────────────────────────────────────
テクニック 7: モデルを目的に合わせて切り替える
────────────────────────────────────────────────────────────────────────────────

  すべてのタスクにデフォルトモデルを使う必要はありません。

  タスク別推奨モデル（このプロジェクトの場合）:

  タスク                              推奨モデル
  ──────────────────────────────────────────────────────────────────────
  CSVパースロジックのバグ修正          claude-sonnet-4.6（デフォルト）
  大規模リファクタリングの設計         gemini-2.5-pro または opus
  単純なフォーマット変更・定数追加     haiku または gemini-2.5-flash
  データ分析の仮説立案・考察           gemini-2.5-pro（推論が深い）
  ドキュメント生成                    claude-sonnet-4.6 で十分

  サブエージェントのモデル上書き例:
  「worker（model: gemini-2.5-pro）を使って、分類ロジックの
   アーキテクチャを再設計してください。」

PART 5: アンチパターン（やりがちな失敗例）
================================================================================

  ❌ アンチパターン1: コンテキストなしの丸投げ
  「data.js をいい感じに改善して」
  → 何を改善すべきか不明。OB-1 が不要な変更をする可能性がある。

  ❌ アンチパターン2: 複数の無関係なタスクを一度に渡す
  「バグを直して、新機能も追加して、ドキュメントも更新して」
  → タスクが混在してコンテキストが汚染される。1タスク1ターンを基本に。

  ❌ アンチパターン3: バンドル生成を忘れる（このプロジェクト固有）
  「classifyProduct() を修正しました」で止まる
  → bundle-static.js は自動更新されない。
    「実装後 node js/create-bundle.js を実行してください」を必ず添える。

  ❌ アンチパターン4: 画像・動画を直接 read_file で読む
  「screenshot.png を読んで内容を教えて」
  → read_file は画像の内容を正確に解析しない。
    必ず vision-analyzer を経由する。

  ❌ アンチパターン5: セッションをまたいで文脈を引き継がない
  新しいセッションで突然「続きをやって」と言う
  → OB-1 は前のセッションを覚えていない。
    背景・進捗・残タスクを毎セッション冒頭で簡潔に伝える。

  ❌ アンチパターン6: 調査エージェントに実装を期待する
  explore エージェントを呼んでファイルを変更させようとする
  → explore は読み取り専用。変更が必要なら worker を使う。

PART 6: このプロジェクト専用のクイックリファレンス
================================================================================

■ 変更→確認の基本コマンドセット
  実装変更後に毎回実行:
  node js/create-bundle.js
  grep -c "変更した関数名" js/bundle-static.js

■ よく使うファイルと役割
  js/config.js    → 閾値・カテゴリ定義（分類条件の変更はここ）
  js/data.js      → CSV解析・日次データ生成・商品分類ロジック
  js/ui.js        → ビュー表示・ダッシュボード更新
  js/charts.js    → Chart.js グラフ描画関数
  js/main.js      → 初期化・イベントリスナー・ファイルハンドリング

■ 新機能追加時の定型プロンプト末尾
  「実装後、以下を確認してください:
   1. node js/create-bundle.js を実行
   2. grep で [追加した関数名] が bundle-static.js に含まれることを確認
   3. バンドルのファイルサイズ（行数）を表示して大きく増えていないことを確認」

■ 分類ロジックを変更するときのプロンプト枠組み
  「js/data.js の classifyProduct() の [カテゴリX] の条件を変更します。
   【変更前】[現在の条件]
   【変更後】[新しい条件]
   【理由】[なぜ変更するか]
   変更後、サンプルCSV（251101-251130-BusinessReport.csv）を使って
   分類結果の変化を確認してください。」
