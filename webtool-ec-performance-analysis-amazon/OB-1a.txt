1. OB-1 とは
================================================================================

OB-1 は、ソフトウェアエンジニアリングタスクに特化したインタラクティブな
CLI エージェントです。コードの読み書き・実行・Web検索・ブラウザ操作・
画像解析・複数エージェントの協調など、幅広いタスクを自律的にこなします。

特徴:
  - 完全なツールセット（ファイル読書き / シェルコマンド / Web検索 / ブラウザ）
  - 専門サブエージェントへの委譲（並列・直列）
  - スレッドによるセッション管理と記憶の永続化
  - 複数モデルの切り替え対応

2. スラッシュコマンド一覧
================================================================================

  コマンド                     説明
  ─────────────────────────────────────────────────────
  /help [command]              コマンドのヘルプ表示
  /bug                         バグ報告・フィードバック送信
  /plan start <goal>           プランモードで多段階タスクを開始
  /plan status                 実行中プランの進捗確認
  /plan abort                  実行中プランのキャンセル
  /agents list                 利用可能なエージェント一覧表示
  /agents refresh              エージェント定義を再読み込み
  /run <agent> <input>         エージェントを直接実行
  /thread new                  新しいスレッド（会話コンテキスト）作成
  /thread list                 スレッド一覧表示
  /thread switch <id>          スレッドの切り替え
  /thread close <id>           スレッドを閉じる（記憶を保存可）
  /model set <model-id>        セッションのデフォルトモデルを変更
  /model list                  利用可能なモデル一覧表示
  /memory save <thread-id>     スレッドの記憶をディスクに保存
  /memory load <thread-id>     保存済みの記憶を読み込み
  /config show                 現在の設定表示
  /config set <key> <value>    設定値の変更

3. サブエージェント詳解
================================================================================

OB-1 には9つの専門サブエージェントが組み込まれています。
適切なエージェントを選ぶことで、速度・精度・コスト効率が大幅に向上します。

────────────────────────────────────────────────────────────────────────────────
3.1 worker — 万能実装エージェント【最もよく使う】
────────────────────────────────────────────────────────────────────────────────
  用途:  コードの実装・ファイル編集・シェルコマンド実行・Web検索・
         多段階の変更を伴う複雑な実装タスク
  特徴:  - メインエージェントと同等のフルツールセットを持つ
         - 自律的に複数ステップをこなせる
         - main との会話履歴の引き継ぎが不要なタスクに最適
  使い方の例:
    「このファイルのバグを修正して、テストも追加して」
    「新しいAPIエンドポイントを実装して動作確認まで行って」

  ⚠ handoff_to_agent との違い:
    - worker はローカルプロセス内で同期実行
    - handoff_to_agent はリモートサービスへの非同期委譲（ユーザーとの
      直接対話が必要な場合や、完全な会話履歴が必要な場合に使用）

────────────────────────────────────────────────────────────────────────────────
3.2 explore — コードベース探索エージェント
────────────────────────────────────────────────────────────────────────────────
  用途:  ファイルパターン検索、キーワード検索、コードの仕組みの調査
  特徴:  - 読み取り専用（ファイル変更不可）
         - 「quick / medium / very thorough」の精度レベルを指定可能
         - 大規模コードベースの概要把握に最適
  使い方の例:
    「src/components/**/*.tsx のファイルを全て調べて」
    「API エンドポイントがどう実装されているか調査して（very thorough）」

────────────────────────────────────────────────────────────────────────────────
3.3 plan — アーキテクチャ設計・計画エージェント
────────────────────────────────────────────────────────────────────────────────
  用途:  複雑な機能追加・リファクタリング前の設計・計画立案
  特徴:  - 計画を生成するが自動実行はしない
         - 依存関係の特定・実装順序の提案・落とし穴の指摘
         - perspective パラメータで設計方針を指定可能
  使い方の例:
    「認証機能を追加する詳細な実装計画を作って（focus on security）」

  ⚠ プランモード（/plan start）との違い:
    - /plan start は計画を自動実行する
    - plan サブエージェントは計画を返すだけ（実行は別途）

────────────────────────────────────────────────────────────────────────────────
3.4 codebase_investigator — コードベース構造解析エージェント
────────────────────────────────────────────────────────────────────────────────
  用途:  システム全体の依存関係把握・バグの根本原因分析・
         大規模リファクタリングの事前調査
  特徴:  - アーキテクチャマップと実行可能な洞察を返す
         - 曖昧な依頼・バグ調査・システム横断的な問題に最適
  使い方の例:
    「このバグがなぜ起きているか根本原因を調べて」
    「プロジェクト全体のデータフローを説明して」

────────────────────────────────────────────────────────────────────────────────
3.5 general — 汎用探索＋実行エージェント
────────────────────────────────────────────────────────────────────────────────
  用途:  探索しながら変更が必要なタスク・複雑な推論を伴う検索・
         初期戦略が失敗した場合に代替手段を試すタスク
  特徴:  - explore と異なりファイル変更・コマンド実行が可能
         - 複数の戦略を試しながら結果を出す
  使い方の例:
    「このエラーの原因を見つけて修正して（調査しながら変更が必要）」

────────────────────────────────────────────────────────────────────────────────
3.6 browser — ブラウザ操作エージェント
────────────────────────────────────────────────────────────────────────────────
  用途:  Webページのスクリーンショット撮影・UI確認・フォーム入力・
         クリック操作・Webアプリのテスト
  特徴:  - Playwright/Chromium を使用
         - 実際のスクリーンショットを撮影できる唯一のエージェント
         - JavaScript実行・CSS/XPathによる要素抽出対応
  使い方の例:
    「http://localhost:3000 のトップページのスクリーンショットを撮って」
    「ログインフォームに情報を入力して送信して」

────────────────────────────────────────────────────────────────────────────────
3.7 vision-analyzer — 画像・動画解析エージェント
────────────────────────────────────────────────────────────────────────────────
  用途:  .png/.jpg/.gif/.mp4/.webm/.mov ファイルの解析
         スクリーンショットのQA確認・OCRテキスト抽出・UI検証
  特徴:  - 画像・動画ファイルは必ずこのエージェント経由で処理
         - 主コンテキストのトークン節約になる
         - バッチ画像解析・デモ検証・バグのビジュアル調査に最適
  使い方の例:
    「screenshot.png のUIを分析して問題点を指摘して」
    「この動画の10〜30秒間に何が表示されているか教えて」

────────────────────────────────────────────────────────────────────────────────
3.8 web — Web検索・URLフェッチエージェント
────────────────────────────────────────────────────────────────────────────────
  用途:  Web検索・Webページの内容取得・調査・情報収集
  特徴:  - Web検索・URLフェッチは必ずこのエージェント経由で実施
         - 大量のWeb内容が主コンテキストを汚染するのを防ぐ
         - 構造化されたサマリーを返す
  使い方の例:
    「Chart.js 4.4 の最新ドキュメントを調べて」
    「https://example.com の内容を要約して」

────────────────────────────────────────────────────────────────────────────────
3.9 cli_help — OB-1 自身の使い方調査エージェント
────────────────────────────────────────────────────────────────────────────────
  用途:  OB-1 の機能・ドキュメント・現在の設定についての質問
  使い方の例:
    「OB-1 のスレッド管理の使い方を教えて」

4. モデル選択
================================================================================

モデルの指定方法は3通りあります:
  1. コマンドラインフラグ: --model <preset>
  2. 環境変数: OB1_MODEL=<preset>
  3. 設定ファイル: ~/.ob1/config.yaml の default_model フィールド

主要プリセット一覧:
  プリセットキー                説明
  ──────────────────────────────────────────────────────────
  claude-sonnet-4.6             Anthropic Claude Sonnet 4.6（デフォルト）
  opus / opus-4.6               Anthropic Claude Opus（高品質・高コスト）
  haiku                         Claude Haiku（軽量・高速）
  gpt-5.4 / gpt-5.4-pro        OpenAI GPT-5.4 系
  gemini-2.5-pro                Google Gemini 2.5 Pro
  gemini-2.5-flash              Google Gemini 2.5 Flash（高速）
  kimi-k2.5                     Kimi K2.5
  deepseek-r1-free              DeepSeek R1（無料）
  grok-4.1-fast                 Grok 4.1 Fast
  openrouter-free               OpenRouter 無料モデル

使い分けの目安:
  - 通常のコーディング         → claude-sonnet-4.6（デフォルト）
  - 高精度・複雑な推論         → opus / gemini-2.5-pro
  - 高速・軽量タスク           → haiku / gemini-2.5-flash
  - コスト節約                → deepseek-r1-free / openrouter-free

5. スレッド管理とセッション継続
================================================================================

■ スレッドとは
  会話コンテキストを隔離した単位です。プロジェクトごとにスレッドを
  分けることで、異なる作業が混在せずに済みます。

■ 基本操作
  /thread new           新しいスレッドを作成
  /thread list          スレッド一覧（ID・最終活動日時付き）
  /thread switch <id>   別のスレッドへ切り替え
  /thread close <id>    スレッドを閉じる

■ フォーク（分岐）
  ob1 thread fork <name>
  現在のワークスペース状態（記憶・プラン・変数）をコピーして
  新しいスレッドを作成。別の方向性を試す「仮定実験」に有用。

■ セッション再開
  ob1 thread resume <name>
  保存済みのスレッドを再開。中断したプランの続きも自動的に再開可能。

6. 記憶（Memory）管理
================================================================================

■ ディスク保存
  /memory save <thread-id>
  全会話を ~/.ob1/memory/<thread-id>.json に保存。

■ ユーザーメモリ（グローバル保存）
  「〇〇を覚えておいて」と伝えると save_memory ツールが実行され、
  全ワークスペースをまたいで保持されるユーザー固有の情報として保存。
  例: 好みのコーディングスタイル、よく使うパス、個人設定など。

  ⚠ プロジェクト固有の情報（エントリーポイント、テストコマンド等）は
    グローバル保存には不向き。


================================================================================
7. ベストプラクティス
================================================================================

  ✅ 実装タスクは worker に委譲（メインコンテキストを温存）
  ✅ Web検索・URLフェッチは web エージェント経由（トークン隔離）
  ✅ 画像・動画は vision-analyzer 経由（同上）
  ✅ 複雑な調査は codebase_investigator に先に任せる
  ✅ 長期プロジェクトは定期的に /memory save
  ✅ 無関係な作業は /thread new で別スレッドに分ける
  ✅ 本番コード変更前に /thread fork で安全なコピーを作る
  ✅ ファイル拡張子は glob で確認してから操作
  ✅ コミットは明示的に「コミットして」と伝えた場合のみ実行
  ✅ エージェント定義変更後は /agents refresh を忘れずに
