OB-1 コーディング最大活用 Tips
================================================================================
作成日: 2026年3月12日

【1】指示の書き方
================================================================================

■ 構造化して伝える
  - 「ゴール → 制約 → 背景」の順で書く（逆順は精度が落ちる）
  - 箇条書き・【】ラベルで構造を明示する（OB-1 は構造化入力に強い）
  - 長い指示は番号付きリストで分割する
  - 一文に複数の指示を詰め込まない（1指示1文が理想）

■ 具体的なシンボルを使う
  - ファイル名・関数名・クラス名・変数名を必ず明記する
  - 「いい感じに直して」ではなく「js/data.js の classifyProduct() の〇〇を修正」
  - 行番号やエラーメッセージはそのままコピペして貼る
  - 対象の CSVカラム名・APIのフィールド名など固有名詞を省略しない

■ スコープを明示する
  - 【やること】と【やらないこと】を必ずセットで書く
  - 触ってほしくないファイルを明示する（例：「main.js は変更しないこと」）
  - 変更してよい範囲を関数単位・ファイル単位で指定する
  - 「最小限の変更で」と添えると既存コードへの影響を抑えられる

■ 完了条件を定義する
  - 「〇〇が動作すれば完了」ではなく検証手順まで書く
  - 例：「bundle-static.js に createEffectChart の文字列が含まれること」
  - 例：「node analyze.js を実行してエラーなく終了すること」
  - 例：「ブラウザで〇〇タブを押して結果が表示されること」

■ 「なぜ」を添える
  - 変更の理由を添えると OB-1 が文脈に合った実装を選ぶ
  - 理由があれば代替案の提案も的確になる
  - 例：「理由：実運用で〇〇の誤判定が多発しているため閾値を調整したい」

【2】サブエージェントの使い分け
================================================================================

■ worker を使うべき場面
  - コードを書く・ファイルを変更する・コマンドを実行するすべてのタスク
  - 3ファイル以上を横断する変更
  - 実装→テスト→動作確認まで一連でこなしてほしいとき
  - メインの会話コンテキストを温存したいとき

■ explore を使うべき場面
  - 変更前に「どうなっているか」を調べたいとき（読み取り専用）
  - ファイルパターン検索（例：「src/**/*.js にある関数を全部列挙して」）
  - キーワードが複数ファイルに散在していてどこを見ればよいか不明なとき
  - 「quick / medium / very thorough」で調査深度を指定できる

■ codebase_investigator を使うべき場面
  - プロジェクトに初めて触れるとき（全体構造の把握）
  - バグの根本原因が不明で複数ファイルを横断している可能性があるとき
  - 大規模リファクタリング前の影響範囲調査
  - 「このモジュールはどこから呼ばれているか」など依存関係の追跡

■ plan を使うべき場面
  - 実装前に設計・順序・依存関係を整理したいとき
  - 実装計画を確認してから worker に渡すワークフローで使う
  - perspective パラメータで「パフォーマンス重視」「最小変更」など方針を指定

■ general を使うべき場面
  - 調査しながら変更も必要な複合タスク（explore と worker の中間）
  - 初回の戦略が失敗したら代替案を自動的に試してほしいとき
  - 調査結果に応じて動的に実装内容を決めてほしいとき

■ web を使うべき場面（必須経由）
  - 外部ライブラリのドキュメント・APIリファレンスを調べるとき
  - StackOverflow・GitHub Issues を参照したいとき
  - 最新の仕様・バージョン情報を確認したいとき
  - 直接呼ばず必ず web エージェント経由にする（トークン汚染を防ぐ）

■ browser を使うべき場面
  - 実装後の UI 確認・スクリーンショット撮影
  - フォーム入力・ボタンクリックなどのインタラクションテスト
  - localhost の動作確認（http://localhost:8000 など）
  - 実際の画面描画を伴う確認は唯一このエージェントのみ可能

■ vision-analyzer を使うべき場面（必須経由）
  - .png / .jpg / .gif / .mp4 / .webm などすべての視覚ファイル
  - スクリーンショットの UI 確認・OCR テキスト抽出
  - グラフ・チャートの内容が正しく描画されているかの確認
  - read_file で画像を読もうとしない（内容を正確に解析できない）

【3】コード実装のTips
================================================================================

■ 実装前の準備
  - 既存コードを先に読ませてから実装指示を出す（スタイル・規約の把握）
  - 類似機能の実装例をファイル名で示す（「charts.js の createAnomalyChart に倣って」）
  - 使用するデータ構造の型・形式を事前に伝える
  - ライブラリのバージョンを明示する（「Chart.js 4.4.1 を使用」）

■ 実装指示の粒度
  - 大きな機能は「設計 → 実装 → 確認」の3ターンに分ける
  - 1ターンで任せる変更は3ファイル以内を目安にする
  - 変更量が多いときは「まずAだけ実装して、確認後にBに進んで」と段階的に指示
  - 「実装しながら気づいた問題は実装を止めて先に報告して」と添えると安全

■ テストの扱い
  - 「実装後にテストも追加して」と最初から指示に含める
  - テストフレームワークを明示する（「Jest で」「pytest で」など）
  - テストを先に書かせてから実装させる（TDD スタイル）もOB-1 は対応可能
  - 「この関数のユニットテストを書いて、その後で実装して」

■ リファクタリング
  - 「動作を変えずに〇〇をリファクタリングして」と明示する
  - 変更前後で同じテストが通ることを確認させる
  - 「1ステップずつ変更して、各ステップで動作確認して」と安全策を取る
  - グローバル変数の削減・関数の分割など目的を具体的に書く

■ デバッグ
  - エラーメッセージ・スタックトレースを全文貼る（省略しない）
  - 再現手順を「1. 〇〇する 2. 〇〇する 3. エラーが出る」の形で書く
  - 「どのファイルの何行目か」が分かっていれば先に伝える
  - 「このエラーが出る前後の console.log を追加して原因を特定して」も有効

【4】コンテキスト管理
================================================================================

■ コンテキストを温存する
  - 実装タスクは worker に委譲してメインの会話を軽量に保つ
  - Web 検索・画像読み込みは専用エージェント経由で隔離する
  - ファイルを大量に読ませるときも explore / codebase_investigator に任せる
  - 「全ファイルを読んで」より「〇〇に関係するファイルだけ読んで」

■ セッションの切り替えタイミング
  - タスクが完全に切り替わったら /thread new で別スレッドを作る
  - 1セッションで扱うファイルは10個以下を目安にする
  - エラーが続いて迷走してきたら一度セッションをリセットして整理する
  - 「前の話は一旦忘れて、この問題だけに集中して」も有効

■ 長期プロジェクトの管理
  - 毎回のセッション冒頭に「背景・現状・今日のゴール」を3行で伝える
  - セッション終了時に「今日の成果・残タスク・次のステップ」を要約させる
  - 重要な決定事項（設計方針・命名規則など）はファイルに書き出して保存
  - /memory save でセッションをディスクに保存して再開時に /memory load

■ 情報の渡し方
  - ファイルパスを正確に書く（相対/絶対どちらかに統一）
  - コードの断片より「ファイル名と関数名」を渡す方が OB-1 が自分で読める
  - 「〇〇.txt に書いてある仕様に従って」でファイル参照を指示できる
  - 大量のコードを貼るより「js/data.js の100〜150行目」のように範囲指定

【5】モデルの選び方
================================================================================

■ デフォルト（claude-sonnet-4.6）でよい場面
  - 通常のコード実装・バグ修正・リファクタリング
  - ドキュメント生成・コメント追加
  - 単純なファイル操作・フォーマット変換
  - 日常的なコーディングタスクの9割はこれで十分

■ 高性能モデル（opus / gemini-2.5-pro）が必要な場面
  - 複雑なアーキテクチャ設計・大規模リファクタリングの計画
  - 根本原因が不明な難しいバグの調査
  - 多くのファイルをまたぐ複雑な依存関係の分析
  - 高度な推論・複数の選択肢の比較検討

■ 軽量モデル（haiku / gemini-2.5-flash）でよい場面
  - 定数の変更・フォーマット修正・ファイル名変更などの単純作業
  - 繰り返し実行する自動化タスク（コスト削減）
  - テキストの整形・翻訳・要約

■ モデルをサブエージェントに個別指定する
  - worker のタスクに「model: gemini-2.5-pro」を指定して重い処理を任せる
  - explore・codebase_investigator にはデフォルトで十分なことが多い
  - コスト意識がある場合、軽量タスクのサブエージェントに haiku を指定する

【6】安全・品質を保つTips
================================================================================

■ 変更前の安全策
  - 大きな変更の前に /thread fork でブランチを作ってから実験する
  - 「変更前にバックアップとして現在のコードを表示して」で確認できる
  - 「まず計画を教えてから、進んでよいか確認して」でワンクッション置く
  - 本番環境のファイルに直接触れる前に git status で状態を確認させる

■ 変更後の検証
  - 「実装後に〇〇で動作確認して」を指示に含める
  - ビルド・コンパイルが必要なプロジェクトは必ずビルドまで含める
  - バンドル生成が必要なプロジェクトはその手順も指示に書く
  - grep・wc などでファイルが期待通り変更されたか数値確認させる

■ コードの一貫性を保つ
  - 「既存のコードスタイルに合わせて」を毎回添える
  - 隣接する類似関数の書き方を参照させる（「createAnomalyChart に倣って」）
  - 命名規則・インデント・コメントスタイルを指示に書く
  - 既存の import 文・依存関係を変えないよう明示する

■ セキュリティ
  - API キー・パスワード・シークレットを会話に貼らない
  - 「環境変数から読み込む形で実装して」を習慣にする
  - 外部入力を受け取る関数には「バリデーションも追加して」と指示する
  - OB-1 はセキュリティ上の問題を検出すると警告してくれる

■ Git 操作の注意点
  - 「コミットして」と明示しない限り OB-1 は git commit しない
  - コミット前に git status・git diff で変更内容を確認させる
  - コミットメッセージの草案を提案させてから承認する
  - git reset --hard など破壊的操作は明示的に指示してから実行

【7】繰り返し作業の効率化
================================================================================

■ 定型タスクのテンプレート化
  - よく使う指示パターンをテキストファイルに保存しておく
  - 「このプロジェクトでは毎回バンドル再生成が必要」などを毎回書かずに済む
    → セッション冒頭に「〇〇.txt の制約に従って」と参照させる
  - プロジェクト固有の注意事項をまとめた RULES.txt を作って毎回渡す

■ 自動化スクリプトを作らせる
  - 繰り返す手作業は「自動化スクリプトを書いて」と頼む
  - 「毎週 CSV を処理して分析レポートを出力するスクリプト」など
  - シェルスクリプト・Python・Node.js どれでも対応可能
  - 「cron で定期実行できる形にして」など追加要件も指定できる

■ ドキュメント生成の活用
  - 「この関数に JSDoc コメントを追加して」
  - 「このモジュールの README を作って」
  - 「変更内容を CHANGELOG に追記して」
  - 「この API の使い方をサンプルコード付きで説明するドキュメントを作って」

■ コードレビューの代替
  - 「このコードの問題点・改善点を指摘して」でレビュー代わりに使える
  - 「セキュリティ・パフォーマンス・可読性の観点でレビューして」
  - 「同じ処理をより簡潔に書く方法はあるか」で代替案を出させる
  - 「このロジックにバグが潜んでいる可能性はあるか」で潜在バグを探させる

【8】このプロジェクト（Amazon ASIN Tracker）固有のTips
================================================================================

■ バンドル生成を常に意識する
  - js/*.js を変更したら必ず「node js/create-bundle.js を実行して」と指示に含める
  - 変更が反映されているか grep で確認させる習慣をつける
    → 例：「grep -c "createEffectChart" js/bundle-static.js」
  - バンドル後のファイルサイズ（wc -l）が極端に変わっていないか確認する

■ 分類ロジックを変更するときの注意
  - js/config.js の THRESHOLDS 定数を変えると全カテゴリに影響が出る
  - 変更前に「現在の閾値で何件がどのカテゴリに分類されているか確認して」
  - 変更後はサンプル CSV で分類結果の変化を必ず確認させる
  - classifyProduct() の優先順位（カテゴリ I が最優先）を崩さないよう明示する

■ IndexedDB を変更するときの注意
  - db.js のスキーマを変えると DB_VERSION を上げる必要がある（config.js）
  - バージョンを上げないと既存データとの互換性問題が起きる
  - 「スキーマ変更時は CONFIG.DB_VERSION をインクリメントして」と指示する

■ 日次データ・週次データの混同を防ぐ
  - weeklyData は変数名だが実体は「日次データの Map」（変数名が誤解を招く）
  - dailyData がフラットな配列、weeklyData が Map<date, stats> の構造
  - カードに表示する値は「dailyData の直近30日集計」が正しい
  - 「weeklyData の最新1件」は日割りの小数値になるため UI 表示に使わない

■ 動作確認のコマンドセット
  - ローカルサーバー起動：python3 -m http.server 8000
  - バンドル再生成：node js/create-bundle.js
  - 反映確認：grep -c "関数名" js/bundle-static.js
  - ファイルサイズ確認：wc -l js/bundle-static.js
