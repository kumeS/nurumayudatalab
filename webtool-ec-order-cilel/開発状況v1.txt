1. プロジェクト概要
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CiLEL（仕入れ代行サービス）の注文シート（FT形式 XLSX）および
作業明細シート（WJFT形式 XLSX）を読み込み、商品情報と作業内容を
一覧表示するブラウザ完結型の単一ページWebアプリ。

サーバー不要。データはすべてクライアントサイド（IndexedDB / localStorage）に保存。

主な目的:
  - 複数回のCiLEL注文履歴をまとめて可視化
  - FT と WJFT の紐付けによる作業内容の把握
  - 再注文用CSVの作成・ダウンロード

依存ライブラリ (CDN):
  - SheetJS (xlsx.js) 0.18.5  — XLSXパース
  - JSZip 3.10.1              — XLSXから埋め込み画像を抽出

3. 動作挙動 ― データフローとループの仕組み
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【起動時フロー】
  app-file-input.js
    → ensurePersistentStorage()  IndexedDB利用可否を確認、不可ならlocalStorageへフォールバック
    → initDB()                   IndexedDB('CiLELViewerDB')を開く
                                 objectStore: 'orderData' (ファイルデータ)
                                 objectStore: 'csvDraft'  (CSV一時保存)
    → restoreFromLocalCache()    保存済みデータをallDataに復元 → renderContent()

【ファイルアップロードフロー】
  #dropZone (DragEvent or click → input[type=file])
    → handleFiles(files)          [app-state-core.js]
        ループ: for each file in files
          → FileReader.readAsArrayBuffer(file)
              → XLSX.read(buffer)   SheetJSでXLSXをパース
              → sheet_to_json()     rows配列取得
              → JSZip.loadAsync()   XLSXのzipを開き埋め込み画像を抽出
                  extractImagesFromXlsx():
                    xl/drawings/drawing1.xml → アンカー行番号マッピング
                    xl/drawings/_rels/drawing1.xml.rels → 画像ファイルパス取得
                    xl/media/* → Blobとして読み込み → data:URL変換
              → detectFileType()    'ft' または 'wjft' を判定
                  判定ロジック: ファイル名が'WJFT'で始まる、
                  または先頭20行に '作業内容' + '番号' が存在すれば WJFT
              → processData() / processWjftData()  データモデル構築
              → saveDataToDB()       IndexedDBに保存
              → renderContent()      UIを再描画

【FTデータ処理 (processData)】
  - ヘッダー行を走査し normalizeHeaderText() でホワイトスペースを除去して列を特定
  - 各データ行をItemオブジェクトに変換
  - 為替レート: CNY→JPY を全行の (JPY単価 / CNY単価) の平均で算出
  - extractOrderInfo(): 注文番号、注文日、合計金額、送料情報等をメタデータとして抽出
  - 結果: allData[fileName] = { items[], orderMeta, exchangeRate, fileType:'ft' }

【WJFTデータ処理 (processWjftData)】
  - ヘッダー行探索: 0〜19行を走査し '作業内容' '番号' を含む行をヘッダー行と判定
  - 複数行にわたる商品の結合ループ:
      現在行が「番号あり」→ 新商品開始
      現在行が「番号なし」→ 前商品のworkItemsに追加行をpush
  - workMeta: FT番号、顧客名、注文日、作業日、検品費等を先頭ブロックから抽出
  - 結果: allData[fileName] = { items[], workMeta, fileType:'wjft' }

【レンダリングフロー】
  renderContent() → currentTab に応じて分岐
    'all'     → 全ファイルのitemsを結合（mergeAllDuplicates が true なら重複統合）
                → productSortMode に従いソート
                → renderProductCard() × n 個 → #productGrid に注入
    'summary' → renderSummaryContent()
                → ファイル一覧テーブル生成
                → サムネイル列: _embeddedImage (data:URL) または photo (URL)
                → 統計カード更新
    fileName  → そのファイルが FT なら renderProductCard() × n
                WJFTなら renderWjftContent()

【重複統合ロジック (全体まとめタブ)】
  キー: siteUrl + unitPriceCny  (なければ variant + subtotal)
  同一キーを持つ複数アイテムをマージ:
    orderQty, stockQty, shipQty, subtotal → 合算
    unitPriceCny → 加重平均
    variant, arrivalDate, _fileName → 配列としてタグ表示

【WJFT↔FTリンク処理】
  findLinkedWjftData(ftFileName):
    allData の全WJFT fileを走査
    workMeta.ftNumber が ftFileName の FT番号（例: FT6705-00017）に一致するかチェック
  findLinkedWorkItem(ftItem, wjftItems):
    URLベース比較（クエリパラメータを除去して一致判定）
  一致した場合: FT商品カードに作業内容サマリーを追加表示

【CSVエクスポートフロー】
  Step1: 「商品発注CSVファイルを作成する」クリック
    → csvExportSelectMode = true → サムネイル選択モード開始
  Step2: サムネイル（全体集計タブ）をクリックで選択/解除
    → csvExportSelectedImageKeys (Set) に画像キーを追加/削除
  Step3: 「表を作成する」クリック
    → resolveSelectedItemsFromImageKeys()
        選択キーから該当itemsを特定するロジック:
          _embeddedImage (data:URL) と imageKey の照合
          photo URL と imageKey の照合
    → showCsvExportTablePopup()
        ヘッダー行 + データ行の編集可能テーブル（contenteditable）を生成
        「サイズ・カラー・各個数など」「納品指示」は改行保持 (white-space: pre-wrap)
  Step4: テーブル編集（contenteditable セル直接編集）
  Step5: 「一時保存」クリック
    → readTableEditedRows() でDOM から現在の編集内容を読み取り
    → saveCsvDraftToDB() でIndexedDBの 'csvDraft' ストアに保存
        保存内容: { selectedImageKeys[], format, headers[], editedRows[][], timestamp }
  Step6: 「CSVエクスポート」クリック
    → generateCsvFromSelectedItems() → BOM付きUTF-8 CSVを生成
    → downloadCsv() → Blobダウンロード
        ファイル名: export_cilel_csv_YYYY-MM-DD-HHMM.csv

【一時保存の復元フロー】
  起動時: initDB() → loadCsvDraftFromDB()
  csvDraftData が存在する場合:
    全体集計タブに「一時保存から表を作成する (日時) ✎」ボタンを表示
  クリック: showCsvExportTablePopup(csvDraftData) で保存済み状態を復元

【ステータスポップアップ】
  発送ステータス（'発送1', '次回発送', '別途発送'）クリック
    → showStatusPopup(statusText, element) → #statusPopup を配置・表示
  スクロール/リサイズ/他クリックで自動非表示

【WJFTタブ インタラクション】
  作業行クリック: ハイライトON/OFF → localStorage('wjft-highlight-{fileName}-{rowId}')
  コメントボタン: prompt() ダイアログ → localStorage('wjft-comment-{fileName}-{rowId}')
  次回起動時: renderWjftContent() 内でlocalStorageから復元・再表示

4. 実装済みだが完成度が低い箇所
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【4-1】CSVポップアップ「一時保存」初回保存バグ ★★★（最重要）
  場所: app-render.js (btnSaveDraftPopup のクリックハンドラ)
  症状: テーブル編集後に「一時保存」を押しても編集内容が保存されない。
        2回目以降は正常に保存される。
  原因: readTableEditedRows() がDOM変更のコミット前に呼ばれている可能性。
        または contenteditable の blur イベント前に読み取りが走るタイミング問題。
  対策案: innerHTML/innerText の読み取り前に setTimeout(0) または
          activeElement.blur() を挟んでDOMをフラッシュする。

【4-2】「一時保存の表を削除する」後もボタンが残るバグ ★★★（最重要）
  場所: app-render.js / app-csv-export.js (deleteCsvDraftFromDB の後処理)
  症状: 削除ボタン押下後も「一時保存から表を作成する」ボタンが消えない。
  原因: deleteCsvDraftFromDB() が非同期で csvDraftData = null をセットする前に
        renderSummaryContent() が走っている、またはそもそも再レンダリングが
        呼ばれていない。
  対策案: 削除完了のコールバック内で明示的に csvDraftData = null → renderSummaryContent()。

【4-3】window.confirm() / window.prompt() の使用 ★★
  場所: app-state-extra.js (ファイル削除確認)、app-helpers.js (コメント入力)
  問題: 一部の埋め込みブラウザ環境（例: ChatGPT 内ブラウザ）では
        これらのネイティブダイアログが動作しない。
  対策案: カスタムモーダルダイアログに置き換える（UI実装コストは小）。
  bug_report.txt にも記録済みだが未対応。

【4-4】「商品を追加する」ポップアップのNo列自動連番なし ★
  場所: app-render.js (showProductPickerForTable / 行削除処理)
  問題: 行削除後に「No」列の番号が再採番されない。
        例: 1,2,3 → 2を削除すると 1,3 のままになる。
  対策案: 行追加・削除のたびに No 列を再計算するユーティリティを呼ぶ。

【4-5】WJFT コメント機能の prompt() 依存 ★
  場所: app-helpers.js (WJFTコメントボタンハンドラ)
  問題: コメント入力に prompt() を使用。
        既存コメントの初期値がセットされているかが不明確。
        インライン編集UIの方がUX向上につながる。

【4-6】WJFTメタデータ抽出の部分未実装 ★
  場所: app-state-extra.js (processWjftData 内 行0〜の走査ブロック)
  問題: コードに「注文日/作業日ラベルをスキャンする」ループの外枠だけ存在し、
        内部処理が空のまま。実際の抽出は別のロジックで行われているが、
        コードの意図が不明瞭で保守が難しい。

【4-7】FT↔WJFT リンクの精度 ★
  場所: app-state-extra.js (findLinkedWorkItem)
  問題: URLのクエリパラメータ除去だけで照合しているため、
        同一ショップの別商品が誤マッチするケースがある。
        商品番号（no）も組み合わせた照合には対応していない。

【4-8】localStorageフォールバックの不完全性 ★
  場所: app-state-core.js
  問題: 埋め込み画像（data:URL）はサイズが大きく（数MB）、
        localStorageの容量上限（約5MB）をすぐに超える。
        IndexedDB利用不可の環境では画像なしで動作する旨の
        ユーザーへの通知が存在しない。

【4-9】Python スクリプトの未統合 ★
  場所: py/analyze_wjft.py
  現状: WJFTのExcel構造解析用スクリプトが存在するが、
        Webアプリとは完全に独立しており、開発時のデバッグ専用。
        ドキュメントもなく利用方法が不明。

IndexedDB: 'CiLELViewerDB'
  objectStore: 'orderData'
    key: fileName (string)
    value: { items[], orderMeta or workMeta, exchangeRate, fileType }
  objectStore: 'csvDraft'
    key: 'draft'
    value: { selectedImageKeys[], format, headers[], editedRows[][], timestamp }

localStorage:
  'CiLELViewerCache'             — allData 全体のスナップショット（フォールバック用）
  'CiLELViewerSummarySortMode'   — 全体集計タブのソートモード
  'CiLELViewerMergeAllDuplicates'— 重複統合トグル状態
  'wjft-highlight-{file}-{id}'   — WJFT作業行ハイライト状態
  'wjft-comment-{file}-{id}'     — WJFT作業行コメント
