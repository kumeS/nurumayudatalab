■ プロジェクト概要
────────────────────────────────────────────────────────────────────────────────
  名称: ぬるっと翻訳・校閲ツール
  目的: LLM (大規模言語モデル) を活用した日英間テキスト翻訳・校閲Webツール
  技術: HTML5 + CSS3 + Vanilla JavaScript (フロントエンドのみ、SPA構成)
  バックエンド: Cloudflare Workers (nurumayu-worker.skume-bioinfo.workers.dev)
  使用LLM: meta-llama/Llama-4-Maverick-17B-128E-Instruct-FP8 (固定)
  開発期間: 2025年5月18日 〜 現在 (約10ヶ月、257コミット)
  ファイル構成: index.html (579行), app.js (642行), workertest.sh, README.md
  合計コード量: 約1,221行

================================================================================
■ 開発進捗・完成度・機能性の総括
================================================================================

【完成度: 約70%（動作するプロトタイプ段階）】

本ツールは、基本的な翻訳・校閲機能が一通り動作する状態に到達しており、
フロントエンドUIとLLM API連携の中核部分は実装済みである。
4つの処理モード（日→英翻訳、英→日翻訳、日本語校閲、英語校閲）が
すべて利用可能であり、結果表示・コピー・読み上げなどの付加機能も備えている。

しかし、以下の観点で「製品レベル」には達していない:
- ストリーミング応答が断念され、非ストリーミングモードでの運用
- LLMモデルが1つに固定（モデル選択UIは存在するが無効化）
- テストコード・自動テストが一切存在しない
- エラーハンドリングが最低限（タイムアウト制御、リトライなし）
- セキュリティ対策（入力サニタイズ、レート制限等）が不足

■ 動作挙動の詳細（ステップごとのフロー）
================================================================================

1. 初期化フェーズ (DOMContentLoaded)
   ├── localStorage から前回の入力テキストを復元
   │   └── 特定の誤文字列 ("This will ensure that...") が保存されていた場合は除去
   ├── タブの初期状態設定 (デフォルト: jaen = 日本語→英語)
   ├── 各ボタン・イベントリスナーの登録
   └── 使い方ガイドのアコーディオン制御を初期化

2. モード選択フェーズ (タブクリック時)
   ├── 全タブから active クラスを除去
   ├── 選択タブに active クラスを付与 + インラインスタイルで視覚更新
   ├── currentMode 変数を更新 (jaen / enja / jajarev / enrev)
   └── 結果エリアのみクリア（入力テキストは保持）

3. 実行フェーズ (runLLM関数)
   ├── 入力バリデーション
   │   ├── 空入力チェック → alert表示
   │   ├── 日本語校閲モード: 日本語文字を含まない場合はエラー表示
   │   └── 英語校閲モード: 日本語文字を含む場合はエラー表示
   ├── ローディングインジケーター表示
   ├── createMessages() でモード別プロンプトを構築
   └── fallbackNonStreamingAPI() を呼び出し

4. プロンプト構築 (createMessages関数) — 各モードのプロンプト詳細
   ┌─────────────────────────────────────────────────────────────────────┐
   │ [jaen] 日本語→英語翻訳                                              │
   │ system: 「翻訳エンジンとして動作。挨拶・説明・コメントを排除。       │
   │          英訳のみ返す。完全翻訳を指示。思考過程出力を禁止。」        │
   │ user:   「以下の日本語文を英語に翻訳してください...『{入力}』」      │
   ├─────────────────────────────────────────────────────────────────────┤
   │ [enja] 英語→日本語翻訳                                              │
   │ system: 「翻訳エンジンとして動作。日本語訳のみ返す。                 │
   │          部分訳・省略禁止。思考過程出力禁止。」                       │
   │ user:   「以下の英文を日本語に翻訳してください...『{入力}』」        │
   ├─────────────────────────────────────────────────────────────────────┤
   │ [jajarev] 日本語校閲                                                │
   │ system: 「日本語校閲エンジン。3セクション出力を要求:                  │
   │          ① 校閲結果: 文法・語彙・構造・読みやすさの解説               │
   │          ② 変更箇所: diff形式 (- 削除 / + 追加) で3〜5箇所            │
   │          ③ 最終案: 改善後の文章全体」                                 │
   │ user:   入力テキストそのまま                                         │
   ├─────────────────────────────────────────────────────────────────────┤
   │ [enrev] 英語校閲                                                    │
   │ system: 「英語校閲エンジン。日本語で校閲コメントを出力。              │
   │          ① 校閲結果: 文法・語彙・構造・自然さの解説(日本語)           │
   │          ② 変更箇所: diff形式で3〜5箇所                               │
   │          ③ 最終案: 改善後の英語文全体」                               │
   │ user:   入力テキストそのまま                                         │
   └─────────────────────────────────────────────────────────────────────┘

5. API通信フェーズ (fallbackNonStreamingAPI関数)
   ├── POST リクエスト送信
   │   ├── URL: https://nurumayu-worker.skume-bioinfo.workers.dev/
   │   ├── model: Llama-4-Maverick-17B-128E-Instruct-FP8
   │   ├── temperature: 0.7
   │   ├── stream: false
   │   ├── max_completion_tokens: 1500
   │   └── messages: モード別プロンプト配列
   ├── レスポンス解析
   │   ├── data.choices[0].message.content → 標準OpenAI互換形式
   │   └── data.answer → フォールバック形式
   └── エラー時: reviewOutput にエラーメッセージ表示

6. レスポンス処理フェーズ
   ├── cleanResponse(): 翻訳モード時のクリーニング
   │   ├── 引用符の除去 ("", '', 「」, 『』)
   │   └── 不要プレフィックスの除去 ("Translation: ", "翻訳結果: " 等)
   ├── 翻訳モード (jaen/enja):
   │   ├── 先頭欠落チェック (isPossiblyTruncatedResponse)
   │   │   └── 助詞で始まる / 小文字で始まる → 警告表示
   │   ├── 末尾欠落チェック (isPossiblyTruncatedEnglishResponse)
   │   │   └── カンマ/セミコロン終わり / 句読点なし → 警告表示
   │   └── finalOutput に結果を設定
   └── 校閲モード (jajarev/enrev):
       └── processReviewOutput() で構造化処理
           ├── 正規表現で「校閲結果:」「変更箇所:」「最終案:」を分離
           ├── 校閲結果 → HTML整形 (見出し強調、リスト化、キーワード強調)
           ├── 変更箇所 → diff形式をHTML化 (diff-removed / diff-added クラス)
           ├── 最終案 → finalOutput に設定
           │   └── 英語校閲時: 日本語のみの行を除外するフィルタ
           └── フォールバック: 旧2セクション形式にも対応

7. ループ・繰り返し処理の仕組み
   ├── 自動リトライ: なし（1回のAPI呼び出しで完結）
   ├── 反復翻訳ワークフロー: 「入力欄に戻す」ボタンによる手動ループ
   │   └── 最終結果 → 入力欄に転送 → 再実行で改善を重ねる
   └── 入力テキスト自動保存: input イベントごとに localStorage 更新

■ 実装済みだが完成度が低い箇所
================================================================================

1. 【モデル選択UI】 ★完成度: 10%
   - index.html にモデル選択のラジオボタンUIが存在する
   - しかし disabled 属性で無効化されており、1モデル固定
   - app.js 側にモデル選択ロジックが一切存在しない
   - requestData.model がハードコードされている

2. 【ストリーミング応答】 ★完成度: 15%
   - workertest.sh にストリーミングテストスクリプトが残存
   - app.js のコメントに「問題が続いているため非ストリーミングに切替」と記載
   - ストリーミング用のJSコードは削除済み（フォールバックのみ残存）
   - 長文処理時のUX（漸次表示）が提供されていない

3. 【英語翻訳の欠落検出】 ★完成度: 40%
   - isPossiblyTruncatedEnglishResponse() の判定ロジックが粗い
   - 「句読点なしで終わる」条件が広すぎ、正常な翻訳も誤検知する可能性
   - /[^\.\?\!]$/ は数字やカッコで終わる正常文もキャッチしてしまう

4. 【校閲結果のパース処理】 ★完成度: 55%
   - LLMの出力形式がプロンプト通りとは限らず、パース失敗時のフォールバックが弱い
   - 「校閲結果が期待される形式で見つかりませんでした」という汎用メッセージのみ
   - LLMがdiff形式を正確に出力しない場合のハンドリングが不十分
   - XSS対策: LLMレスポンスをinnerHTMLで直接挿入しており危険

5. 【エラーハンドリング全般】 ★完成度: 30%
   - ネットワークタイムアウトの制御がない（fetch にAbortController未使用）
   - APIレート制限やサーバーエラー (5xx) の判別・リトライがない
   - ユーザーへのエラーメッセージが技術的すぎる
   - ローディング表示がエラー時に残り続ける可能性がある
     （catch内でloadingIndicator.classList.remove('active')はあるが不完全）

6. 【タブのスタイル管理】 ★完成度: 60%
   - CSSで.tab.activeスタイルを定義しているにもかかわらず、
     updateTabDisplay()でインラインスタイルを上書きしている（二重管理）
   - デバッグ用console.logが大量に残存している

7. 【レスポンシブデザイン】 ★完成度: 50%
   - 基本的な@media (max-width: 600px)対応はある
   - タブレットサイズ (601px-1024px) の中間ブレークポイントがない
   - タブが4つ並ぶため、小画面で文字が潰れる可能性

■ 未実装だが有用な新機能の提案（短期実装向け）
================================================================================

提案1: 【翻訳履歴機能】 実装難易度: ★★☆☆☆ (2〜3時間)
  概要: 翻訳・校閲の入出力ペアを localStorage に保存し、一覧表示・再利用
  利点: 過去の翻訳結果を参照でき、同じ文章の再翻訳を防止
  実装: localStorage に配列として {mode, input, output, timestamp} を保存
        サイドパネルまたはモーダルで履歴一覧を表示
        履歴項目クリックで入力欄に復元

提案2: 【文字数・トークン数カウンター】 実装難易度: ★☆☆☆☆ (30分〜1時間)
  概要: 入力テキストの文字数・単語数をリアルタイム表示
  利点: max_completion_tokens制限(1500)に対する入力量の目安を提供
        翻訳前後の文量比較ができる
  実装: inputイベントで文字数/単語数をカウントし、入力欄下部に表示

提案3: 【ダークモード切り替え】 実装難易度: ★★☆☆☆ (1〜2時間)
  概要: CSS変数を活用した明暗テーマの切り替え機能
  利点: 夜間作業や目の負担軽減、モダンなUX
  実装: CSS変数のダーク版を定義し、トグルボタンでbodyにクラス切り替え
        localStorage で設定を保持

提案4: 【エクスポート機能（テキスト/Markdown）】 実装難易度: ★★☆☆☆ (1〜2時間)
  概要: 翻訳結果・校閲結果をテキストファイルやMarkdown形式でダウンロード
  利点: 結果の保存・共有が容易になる
  実装: Blob + URL.createObjectURL でダウンロードリンクを生成
        校閲結果はMarkdown形式 (diff含む) で出力

提案5: 【入力言語自動判定とモード推奨】 実装難易度: ★☆☆☆☆ (30分〜1時間)
  概要: 入力テキストの言語を自動判定し、適切なモードを推奨・自動切替
  利点: ユーザーがモードを間違える事故を防止（現在は校閲モードのみ検証あり）
  実装: 既存のcontainsJapaneseChars/containsLatinChars関数を活用
        入力時に言語を判定し、現在のモードと不一致なら通知バーを表示
        ワンクリックで推奨モードに切り替え

提案6: 【APIリクエストのタイムアウト・リトライ機能】 実装難易度: ★★☆☆☆ (1〜2時間)
  概要: AbortControllerによるタイムアウト制御と自動リトライ
  利点: ネットワーク不安定時やAPI一時障害時のUX改善
  実装: fetch にAbortController (30秒タイムアウト) を追加
        失敗時に最大2回リトライ（指数バックオフ）
        リトライ中のUI表示（「再試行中...」）
